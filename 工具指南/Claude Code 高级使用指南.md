### 安全重构

#### 1. 基础解法：分析优先，决策权在你

重构前最常见的问题是"一不小心就改坏了"。解决方案很简单：**让Claude先分析，再提建议，你来决定是否执行**。

**提问示例：**

```
我想重构这个函数，但很担心改坏。能帮我分析一下吗？

[粘贴函数代码]

请不要直接改代码，先告诉我：
1. 这个函数的核心逻辑是什么
2. 有哪些潜在的重构方向
3. 每个方向的风险是什么
```

**关键原则：暂时不要动代码**。让Claude充分分析后，你理解了风险和收益，再决定是否让它执行改动。

---

#### 2. 高级技巧

**技巧1：先写测试，再重构（TDD保护网）**

在重构前为现有代码写测试，这样重构后可以验证行为没变。

```
我想重构这个函数。首先，能帮我写一些测试用例吗？

[粘贴函数代码]

请写出能覆盖以下场景的测试：
- 正常输入
- 边界情况
- 异常情况

不要改函数本身，只写测试。
```

测试写好后，运行测试确保通过，然后开始重构。每次改动后都运行测试，确保行为一致。

**技巧2：小步重构，逐步提交**

不要一次性大改，而是分成多个小步骤，每步都是一个独立的改进。

```
我想分步重构这个函数。第一步，能帮我提取这个逻辑块成为一个独立函数吗？

[粘贴代码]

只做这一个改动，保持其他部分不变。
```

每完成一步，就提交一个commit。这样如果某一步出问题，可以轻松回滚。

**技巧3：用子Agent对比方案**

当有多个重构方案时，让Claude分别实现它们，然后对比优缺点。

```
这个函数有两种重构方案：
1. 使用策略模式
2. 使用工厂模式

能分别给我展示这两种方案的实现吗？然后对比它们的优缺点。
```

这样你可以看到具体代码，而不是抽象的描述，做出更明智的选择。

---

#### 3. 最佳实践

- **重构前先有测试**：没有测试的重构是盲目的。即使是简单函数，也值得写几个测试用例。

- **明确约束条件**：告诉Claude你的约束（比如"不能改API"、"必须保持向后兼容"），这样建议会更贴切。

- **小步前进**：每次改动一个小东西，运行测试，提交。不要试图一次性解决所有问题。

- **让Claude解释每个改动**：不要盲目接受建议。要求Claude解释为什么这样改更好，这样你才能学到东西。

**完整工作流示例：**

```
第一步：分析
"这个函数有什么问题吗？"

第二步：写测试
"能帮我写测试吗？"

第三步：小步改进
"第一步，能提取这个逻辑块吗？"

第四步：验证
"测试还通过吗？"

第五步：下一步
"还有其他改进空间吗？"
```

这个流程虽然看起来慢，但实际上是最快的——因为你不会走弯路，也不会改坏代码。

### 调试Bug

#### 1. 基础解法

当遇到Bug时，最有效的方式是向Claude描述现象，让Claude帮助猜测原因。

**如何描述Bug：**

```
我的应用在用户点击"提交"按钮时崩溃了。
错误信息是：TypeError: Cannot read property 'id' of undefined
这发生在处理表单数据的时候。
```

这样的描述包含了三个关键信息：
- 什么时候发生（用户点击提交按钮）
- 错误信息是什么（TypeError）
- 在哪个流程中（表单数据处理）

**提问示例：**

```
我的React组件在初始化时报错：
Error: setState called on unmounted component

代码是这样的：
useEffect(() => {
  fetchData().then(data => {
    setState(data);  // 这里报错
  });
}, []);

这个组件在某些情况下会快速卸载。
```

**有效描述Bug的要点：**
- 提供完整的错误堆栈跟踪
- 说明复现步骤（如何让Bug出现）
- 提供相关的代码片段
- 说明预期行为 vs 实际行为

---

#### 2. 高级技巧

**技巧1：用Context管理保留调试历史**

在调试复杂Bug时，保留完整的对话历史很重要。使用Claude Code的Context功能：

```
/context add src/components/Form.jsx
/context add src/utils/validation.js
/context add src/hooks/useFormState.js
```

这样Claude能看到完整的代码关系，而不是零散的片段。当你尝试多个假设时，Claude能记住之前的分析。

**技巧2：用指令技巧精准定位问题**

使用结构化的提问方式来精准定位：

```
我的应用在特定条件下出现内存泄漏。

请按以下步骤帮我分析：
1. 首先，列出所有可能导致内存泄漏的原因
2. 然后，根据我的代码特点，排除不可能的原因
3. 最后，给出最可能的3个原因和验证方法

代码片段：[提供代码]
```

这种方式让Claude系统地思考，而不是随意猜测。

**技巧3：用子Agent并行测试多个假设**

当有多个可能的原因时，可以让Claude同时测试多个假设：

```
我的API调用有时成功有时失败。可能的原因有：
1. 网络超时
2. 服务器返回错误
3. 数据格式不匹配

请为每个假设写一个测试用例，我会逐个运行。
```

然后根据测试结果，Claude能快速缩小范围。

---

#### 3. 最佳实践

**如何有效地向Claude描述Bug：**

- 提供最小复现案例（Minimal Reproducible Example）
- 包含完整的错误堆栈跟踪，不要截断
- 说明环境信息（Node版本、浏览器版本等）
- 提供相关的配置文件（package.json、tsconfig.json等）

**提供足够的上下文信息：**

```
不好的描述：
"我的代码不工作"

好的描述：
"在Node 18.0.0 + Express 4.18.0 环境下，
POST /api/users 端点在处理包含特殊字符的用户名时返回500错误。
错误日志显示：SyntaxError: Unexpected token in JSON at position 42"
```

**常见的调试陷阱：**

- 只提供错误信息，不提供代码 → Claude无法定位问题
- 提供过多无关代码 → Claude难以找到关键部分
- 没有说明复现步骤 → Claude无法验证修复
- 隐藏敏感信息导致信息不完整 → 影响诊断准确性

**如何验证修复是否正确：**

修复后，不要只测试一次，要：

1. 测试原始Bug场景是否真的解决了
2. 测试边界情况（空值、极限值、特殊字符等）
3. 测试是否引入了新的Bug（回归测试）
4. 检查性能是否受到影响

```
修复前后对比：
- 原始Bug：用户名包含"@"时崩溃
- 修复后：测试了"@"、"#"、"$"等特殊字符
- 边界测试：空字符串、超长字符串、Unicode字符
- 性能：验证修复没有增加API响应时间
```

## 👥 团队协作场景

### 生成文档

#### 1. 基础解法：让Claude根据代码生成规范文档

最直接的方式是粘贴代码，让Claude生成相应的注释和README。

**提问示例：**

```
请为这个函数生成规范的JSDoc注释：

function calculateDiscount(price, quantity, customerType) {
  if (customerType === 'VIP' && quantity > 10) {
    return price * quantity * 0.85;
  }
  if (quantity > 5) {
    return price * quantity * 0.90;
  }
  return price * quantity;
}

要求：
- 包含参数说明、返回值、使用示例
- 遵循JSDoc格式
```

**生成README示例：**

```
我的项目结构如下：
- /src/api：所有API调用
- /src/components：React组件
- /src/utils：工具函数
- /src/hooks：自定义hooks

能帮我生成一个清晰的README吗？包括项目描述、快速开始、文件结构说明。
```

---

#### 2. 高级技巧

**技巧1：在CLAUDE.md中定义文档规范**

在项目根目录创建或编辑CLAUDE.md，定义你的文档标准：

```
# 项目文档规范

## 代码注释
- 使用JSDoc格式
- 每个公开函数必须有说明
- 复杂逻辑需要行内注释说明"为什么"

## API文档
- 参数类型必须明确
- 返回值要有示例
- 列出所有可能的错误情况

## README规范
- 包括项目目的、快速开始、API文档
- 使用目录和代码示例
```

这样Claude在生成文档时会自动遵循你的规范。

**技巧2：用hooks在保存时自动触发文档更新**

设置编辑器hook（如VS Code的保存触发），当代码改动时自动提示生成/更新文档：

```
监听代码变化 → 触发Claude Code → 检查文档是否需要更新
```

**技巧3：让Claude生成多种格式的文档**

一次性要求Claude生成完整的文档套件：

```
请为这个模块生成以下文档：
1. JSDoc注释
2. 模块级README（说明模块目的、主要函数）
3. API使用指南（包括代码示例）
4. 常见问题解答

代码：[粘贴代码]
```

---

#### 3. 最佳实践

- **先写代码再生成文档**：不要试图在写代码时同步写文档。完成功能后，一次性生成。

- **让Claude保持风格一致**：在CLAUDE.md中定义风格后，Claude生成的所有文档会保持一致。

- **定期更新**：每次代码有重大改动时，重新生成相应的文档，确保文档不会过时。

- **审查和调整**：Claude生成的文档不一定完美，审查后再提交。特别是要确保示例代码能实际运行。

### 理解他人代码库

#### 1. 基础解法：从入口文件开始，让Claude解释架构

最有效的方式是从项目的入口文件开始，让Claude逐步解释整个架构。

**提问示例：**

```
我需要快速理解这个项目的架构。

首先，这是项目的入口文件：
[粘贴 main.js 或 index.js 的内容]

请告诉我：
1. 这个项目的核心目的是什么
2. 主要的模块有哪些
3. 数据流是怎样的
4. 我应该从哪些文件开始深入了解
```

**逐步深入的提问：**

```
现在我想理解认证模块。这是相关的文件：
[粘贴 auth.js、middleware.js 等]

请解释：
- 认证流程是怎样的
- 关键的函数和它们的作用
- 这个模块依赖了哪些其他模块
```

---

#### 2. 高级技巧

**技巧1：让Claude生成「项目地图」**

要求Claude生成一个清晰的项目结构和模块关系图：

```
请为这个项目生成一个"项目地图"，包括：
1. 文件夹结构和每个文件夹的用途
2. 核心模块及其职责
3. 模块之间的依赖关系（用ASCII图表示）
4. 数据流向（从用户输入到输出）

项目结构：
[粘贴 tree 或 ls -R 的输出]
```

**技巧2：用@文件引用来分析特定模块**

在Claude Code中使用@文件引用，让Claude同时看到多个相关文件：

```
@src/api/users.js
@src/models/User.js
@src/middleware/auth.js

这三个文件是如何协作的？请解释数据流。
```

**技巧3：让Claude解释「为什么」而不只是「是什么」**

```
我看到这个函数用了很复杂的递归逻辑。为什么要这样设计？
有没有更简单的方式？这样设计的优缺点是什么？

代码：[粘贴代码]
```

---

#### 3. 最佳实践

- **从整体到局部**：先理解项目的全貌和核心流程，再深入具体模块。

- **关注数据流**：理解数据如何从输入流向输出，这比理解单个函数更重要。

- **提问具体问题**：不要问"这个项目是干什么的"，而要问"用户点击按钮后发生了什么"。

- **记录关键发现**：在理解过程中，让Claude帮你总结关键的设计决策和模块职责。

### Context管理

#### 1. 什么是Context，为什么重要

**Context**是Claude能记住的对话历史。在Claude Code中，你的每条消息、每个文件引用、每个回答都会被保存在Context中。Context的大小直接影响Claude的回答质量：

- **Context充足**：Claude能看到完整的代码关系、之前的分析、你的偏好设置
- **Context混乱**：无关信息堆积，Claude容易给出不相关的建议
- **Context过期**：旧的信息没有更新，Claude基于过时的假设做决策

---

#### 2. 如何有效地使用Context

**技巧1：用`/clear`清空不需要的历史**

当你切换到完全不同的任务时，清空旧的Context：

```
/clear
```

这样Claude不会被之前的讨论干扰。比如从调试Bug切换到写文档时，清空Context能让Claude更专注。

**技巧2：用`@file`引用关键文件而不是复制粘贴**

不要把整个文件内容复制到消息里，而是用`@file`引用：

```
我想理解@src/utils/validation.js中的逻辑
```

这样Claude能看到文件内容，但不会浪费Context空间。

**技巧3：用CLAUDE.md保存项目级别的信息**

把项目的通用信息写在CLAUDE.md中，而不是每次都重复说：

```
# 项目信息
- 使用TypeScript + React
- 代码风格：Prettier + ESLint
- 测试框架：Jest
```

Claude会自动读取这个文件，你不需要每次都解释。

---

#### 3. Context的常见陷阱

**陷阱1：堆积太多无关信息导致回答变差**

如果你在同一个对话中讨论了10个不同的问题，Claude的回答会变得模糊。解决方案：定期清空Context，为新任务开始新对话。

**陷阱2：忘记更新过时的context**

你说"我的项目用React 16"，但实际已经升级到React 18。Claude会基于过时信息给建议。解决方案：定期检查CLAUDE.md，确保信息是最新的。

**陷阱3：跨项目混淆context**

在项目A中讨论了代码风格，然后切换到项目B。Claude可能会混淆两个项目的约定。解决方案：切换项目时用`/clear`清空Context。

### CLAUDE.md定制

#### 1. CLAUDE.md的作用和语法

**CLAUDE.md**是放在项目根目录的特殊文件，Claude Code会在每次对话开始时自动读取它。它的作用是告诉Claude这个项目的规则和偏好。

基本语法很简单，就是Markdown格式的纯文本：

```
# 项目信息
- 使用TypeScript + React 18
- 代码风格：Prettier + ESLint

## 代码约定
- 函数名用camelCase
- 常量用UPPER_SNAKE_CASE
- 组件用PascalCase

## 测试要求
- 所有公开函数必须有单元测试
- 测试框架：Jest
```

Claude会自动理解这些信息，不需要特殊的语法。

---

#### 2. 常见的定制场景

**场景1：定义代码风格和约定**

```
## 代码风格
- 使用2空格缩进
- 字符串用单引号
- 行长限制80字符
- 禁止使用var，只用const/let
```

这样Claude生成的代码会自动遵循这些规则。

**场景2：指定审查标准**

```
## 代码审查标准
- 必须有错误处理
- 性能关键路径需要注释说明复杂度
- 所有API调用必须有超时设置
```

当你让Claude审查代码时，它会按这些标准检查。

**场景3：设置项目特定的术语和缩写**

```
## 项目术语
- API：后端接口
- UI：用户界面
- DAL：数据访问层
- BFF：后端即服务
```

这样Claude不会误解你的缩写。

---

#### 3. 如何写出有效的CLAUDE.md

**原则1：简洁明确，避免冗长**

不要写成一本书。CLAUDE.md应该是快速参考，而不是详细文档。

```
好的：
- 使用TypeScript
- 测试框架：Jest

不好的：
我们的项目使用TypeScript作为编程语言，因为它提供了类型安全...
```

**原则2：包含具体例子**

抽象的规则Claude可能理解不准确。加上例子会更清楚：

```
## 错误处理
✓ 好的例子：
try {
  const data = await fetchData();
} catch (error) {
  logger.error('Failed to fetch', error);
  throw new CustomError('Data fetch failed');
}

✗ 避免：
try {
  const data = await fetchData();
} catch (e) {
  console.log(e);
}
```

**原则3：定期更新**

当项目规则改变时，更新CLAUDE.md。比如从Jest切换到Vitest，要立即更新。

|-----------|------|------|
| `@file` | 快速引用文件 | 在提问中直接引用文件路径，无需复制粘贴 |
| `@folder/` | 引用整个文件夹 | 让Claude阅读文件夹中的所有文件 |
| `/clear` | 清空context | 清除对话历史，开始新的对话 |
| `Shift+Enter` | 换行 | 在输入框中换行，不发送消息 |
| `Ctrl/Cmd+K` | 新对话 | 开始一个新的对话 |
| `Ctrl/Cmd+/` | 快捷键帮助 | 查看所有快捷键 |

### 常见的Prompt模式

**1. 角色扮演**
```
你是一个[角色]，请帮我[任务]
```

**2. 对比分析**
```
对比以下两个方案的优缺点：
方案A：[描述]
方案B：[描述]
```

**3. 分步骤执行**
```
请分3个步骤完成[任务]：
1. [第一步]
2. [第二步]
3. [第三步]
```

**4. 代码审查**
```
@file 请对这个文件做代码审查，重点关注：
- 性能问题
- 安全隐患
- 可读性
```

### 最佳实践速查

- **提供背景信息**：告诉Claude你的项目是什么、用什么技术栈
- **明确约束条件**：「不要改接口」「保持向后兼容」
- **要求具体输出**：「生成单元测试」「用Mermaid图表示」
- **验证理解**：让Claude用你的语言重述理解
- **小步前进**：大任务分解成小任务，逐步完成
- **保存有用的context**：在CLAUDE.md中记录项目级别的信息
