## 🔧 日常编码场景

### 理解陌生代码

#### 1. 基础解法

最直接的方法是让 Claude 逐行解释代码。适合场景：接手遗留代码、阅读开源库、理解同事写的复杂逻辑。

**提问示例：**

```
请逐行解释这段代码的作用，重点说明每个函数的输入输出和副作用。

[粘贴代码]
```

```
这里的 reduce 调用是什么意思？为什么初始值是空对象？
```

```
这段代码的核心逻辑是什么？用一句话总结它在做什么。
```

适用场景：代码片段较短（< 200行）、问题明确、不需要理解跨文件依赖。

---

#### 2. 高级技巧

**技巧1：用 @file 引用建立完整上下文**

当陌生代码涉及多个文件时，单独粘贴片段往往无法理解全貌。使用 `@文件名` 引用让 Claude 同时看到相关文件。

```bash
# 引用多个相关文件
@src/auth/middleware.ts @src/types/user.ts
"这个中间件的鉴权逻辑是怎么工作的？token 是从哪里来的？"

# 引用入口文件理解整体结构
@src/index.ts @src/router.ts
"从入口开始，请梳理一个请求的完整处理链路。"
```

**技巧2：用分层提问管理 Context 复杂度**

面对大型陌生代码库时，采用「先宏观后微观」的分层策略，每次对话聚焦一个层次。

```bash
# 第一轮：理解整体架构（保持 context 干净）
"这个项目的目录结构是怎样的？各个模块分别负责什么？"

# 第二轮（新对话）：深入某个模块
@src/payment/ "支付模块的核心流程是什么？从用户点击支付到回调完成，经过哪些关键步骤？"

# 第三轮：聚焦具体问题
@src/payment/processor.ts "processPayment 函数中，为什么要在第47行做两次重试？这个设计意图是什么？"
```

**技巧3：在 CLAUDE.md 中预设代码库背景**

如果你长期需要理解某个代码库，在项目根目录创建 `CLAUDE.md`，让 Claude 每次都带着正确背景知识读代码。

```markdown
## 项目背景
这是一个电商后台服务，基于 NestJS + TypeORM，数据库是 PostgreSQL。

## 核心概念
- Order：订单实体，状态机驱动（pending → paid → shipped → completed）
- SKU：库存单元，与 Product 是多对一关系
- 所有金额字段单位为「分」，避免浮点数精度问题

## 代码约定
- `*Service` 文件包含业务逻辑
- `*Repository` 文件包含数据库操作
- 错误统一用 `AppException` 抛出，不要用原生 Error

## 注意事项
- legacy/ 目录下是旧代码，逻辑可能有问题，不要作为参考
- 凡是有 TODO(tech-debt) 注释的地方都是已知技术债
```

**技巧4：让 Claude 生成代码地图**

对于完全陌生的大型代码库，先让 Claude 生成一份「代码地图」作为后续提问的索引。

```bash
# 生成函数调用关系
@src/ "请列出这个模块中所有公开函数及其调用关系，用树状结构展示。"

# 识别关键路径
"如果我要修改用户注册逻辑，哪些文件会受到影响？请列出需要关注的文件清单。"

# 生成数据流向描述
"请描述数据从 API 入口到数据库写入的完整流向，标出每个环节做了什么转换。"
```

---

#### 3. 最佳实践

- **先问「做什么」，再问「怎么做」**：先理解函数的业务意图，再理解实现细节。直接问实现细节容易迷失在代码噪音中。

- **带着具体问题读代码**：「这个函数为什么要加锁？」比「解释这段代码」能得到更有价值的回答。泛泛的问题得到泛泛的答案。

- **说出你的困惑点**：「这里用了 Proxy，我不太熟悉这个模式，请结合这段代码解释它解决了什么问题」比「解释 Proxy」更有针对性。

- **顺带问「危险区域」**：理解陌生代码时，附带问「这段代码有哪些潜在风险或容易踩坑的地方？」提前了解雷区。

- **用角色扮演获得更深层解释**：「请以这段代码的原作者身份解释，你当时为什么这样设计，有没有考虑过其他方案？」有时能得到更有洞察力的分析。

- **常见陷阱——Context 污染**：在同一对话里理解多段无关代码，会导致 Claude 混淆上下文。理解不同模块时，开启新会话保持 context 干净。

- **常见陷阱——过度信任解释**：Claude 可能对代码意图做出「合理但错误」的推断，特别是当代码本身有 bug 或违反常规模式时。对关键逻辑，用测试或实际运行验证 Claude 的解释。

### 安全重构

#### 1. 基础解法：分析优先，决策权在你

重构前最常见的问题是"一不小心就改坏了"。解决方案很简单：**让Claude先分析，再提建议，你来决定是否执行**。

**提问示例：**

```
我想重构这个函数，但很担心改坏。能帮我分析一下吗？

[粘贴函数代码]

请不要直接改代码，先告诉我：
1. 这个函数的核心逻辑是什么
2. 有哪些潜在的重构方向
3. 每个方向的风险是什么
```

**关键原则：暂时不要动代码**。让Claude充分分析后，你理解了风险和收益，再决定是否让它执行改动。

---

#### 2. 高级技巧

**技巧1：先写测试，再重构（TDD保护网）**

在重构前为现有代码写测试，这样重构后可以验证行为没变。

```
我想重构这个函数。首先，能帮我写一些测试用例吗？

[粘贴函数代码]

请写出能覆盖以下场景的测试：
- 正常输入
- 边界情况
- 异常情况

不要改函数本身，只写测试。
```

测试写好后，运行测试确保通过，然后开始重构。每次改动后都运行测试，确保行为一致。

**技巧2：小步重构，逐步提交**

不要一次性大改，而是分成多个小步骤，每步都是一个独立的改进。

```
我想分步重构这个函数。第一步，能帮我提取这个逻辑块成为一个独立函数吗？

[粘贴代码]

只做这一个改动，保持其他部分不变。
```

每完成一步，就提交一个commit。这样如果某一步出问题，可以轻松回滚。

**技巧3：用子Agent对比方案**

当有多个重构方案时，让Claude分别实现它们，然后对比优缺点。

```
这个函数有两种重构方案：
1. 使用策略模式
2. 使用工厂模式

能分别给我展示这两种方案的实现吗？然后对比它们的优缺点。
```

这样你可以看到具体代码，而不是抽象的描述，做出更明智的选择。

---

#### 3. 最佳实践

- **重构前先有测试**：没有测试的重构是盲目的。即使是简单函数，也值得写几个测试用例。

- **明确约束条件**：告诉Claude你的约束（比如"不能改API"、"必须保持向后兼容"），这样建议会更贴切。

- **小步前进**：每次改动一个小东西，运行测试，提交。不要试图一次性解决所有问题。

- **让Claude解释每个改动**：不要盲目接受建议。要求Claude解释为什么这样改更好，这样你才能学到东西。

**完整工作流示例：**

```
第一步：分析
"这个函数有什么问题吗？"

第二步：写测试
"能帮我写测试吗？"

第三步：小步改进
"第一步，能提取这个逻辑块吗？"

第四步：验证
"测试还通过吗？"

第五步：下一步
"还有其他改进空间吗？"
```

这个流程虽然看起来慢，但实际上是最快的——因为你不会走弯路，也不会改坏代码。

### 代码审查

#### 1. 基础解法

最直接的方式是将代码粘贴给Claude，并明确指出审查的重点：

\`\`\`
我想让你审查这段代码，重点关注：
1. 是否有明显的性能问题
2. 错误处理是否完善
3. 代码可读性

代码如下：
[粘贴代码]
\`\`\`

或者针对特定方面的审查：

\`\`\`
请审查这个函数的安全性，特别是输入验证和边界情况处理。
\`\`\`

**重点关注的方面**：
- **性能**：算法复杂度、不必要的循环、内存泄漏
- **安全**：输入验证、SQL注入、XSS、权限检查
- **可读性**：命名、注释、代码结构
- **可维护性**：重复代码、魔法数字、依赖关系
- **错误处理**：异常捕获、边界情况、日志记录

---

#### 2. 高级技巧

**技巧1：用CLAUDE.md定制审查标准**

在项目根目录创建 \`CLAUDE.md\`，定义团队的代码审查标准：

\`\`\`markdown
# 代码审查标准

## 必检项
- [ ] 所有公开API都有文档注释
- [ ] 错误处理完善（try-catch或Result类型）
- [ ] 没有TODO/FIXME注释
- [ ] 单元测试覆盖率 > 80%

## 性能检查
- [ ] 没有N+1查询问题
- [ ] 没有不必要的深拷贝
- [ ] 循环内没有重复计算

## 安全检查
- [ ] 用户输入都经过验证
- [ ] 敏感信息不在日志中
- [ ] 没有硬编码的密钥
\`\`\`

然后告诉Claude：

\`\`\`
请按照CLAUDE.md中定义的标准审查这段代码。
\`\`\`

**技巧2：用hooks自动化审查**

在 \`.git/hooks/pre-commit\` 中集成Claude审查：

\`\`\`bash
#!/bin/bash
# 获取staged的代码
git diff --cached > staged.diff

# 调用Claude Code进行审查
claude code review staged.diff --check-security --check-performance

# 如果审查失败，阻止提交
if [ $? -ne 0 ]; then
  echo "代码审查未通过，请修复问题后重试"
  exit 1
fi
\`\`\`

**技巧3：针对性审查**

根据不同场景进行专项审查。

**安全审查：**
\`\`\`
请从安全角度审查这段代码，特别检查：
- 输入验证
- 认证/授权
- 数据加密
- SQL注入风险
\`\`\`

**性能审查：**
\`\`\`
请分析这段代码的性能，包括：
- 时间复杂度
- 空间复杂度
- 是否有可优化的地方
- 建议的优化方案
\`\`\`

**可读性审查：**
\`\`\`
请评估代码的可读性和可维护性：
- 变量/函数命名是否清晰
- 是否需要更多注释
- 代码结构是否合理
- 是否有重复代码可以提取
\`\`\`

---

#### 3. 最佳实践

**审查的关键点**：
- 从业务逻辑开始，再看技术细节
- 关注边界情况和异常路径
- 检查是否遵循项目的编码规范
- 验证是否有测试覆盖
- 确认文档和注释的准确性

**常见问题类型**：
- 空指针异常：未检查null/undefined
- 资源泄漏：文件/连接未正确关闭
- 并发问题：竞态条件、死锁
- 性能问题：不必要的循环、重复计算
- 安全漏洞：输入验证不足、权限检查缺失

**如何有效地与Claude协作进行审查**：
- 提供完整的上下文（相关的依赖、配置、测试）
- 明确审查的目标和优先级
- 对Claude的建议进行讨论，而不是盲目接受
- 将审查反馈转化为团队的编码规范
- 定期更新审查标准，基于实际发现的问题

### 调试Bug

#### 1. 基础解法

当遇到Bug时，最有效的方式是向Claude描述现象，让Claude帮助猜测原因。

**如何描述Bug：**

```
我的应用在用户点击"提交"按钮时崩溃了。
错误信息是：TypeError: Cannot read property 'id' of undefined
这发生在处理表单数据的时候。
```

这样的描述包含了三个关键信息：
- 什么时候发生（用户点击提交按钮）
- 错误信息是什么（TypeError）
- 在哪个流程中（表单数据处理）

**提问示例：**

```
我的React组件在初始化时报错：
Error: setState called on unmounted component

代码是这样的：
useEffect(() => {
  fetchData().then(data => {
    setState(data);  // 这里报错
  });
}, []);

这个组件在某些情况下会快速卸载。
```

**有效描述Bug的要点：**
- 提供完整的错误堆栈跟踪
- 说明复现步骤（如何让Bug出现）
- 提供相关的代码片段
- 说明预期行为 vs 实际行为

---

#### 2. 高级技巧

**技巧1：用Context管理保留调试历史**

在调试复杂Bug时，保留完整的对话历史很重要。使用Claude Code的Context功能：

```
/context add src/components/Form.jsx
/context add src/utils/validation.js
/context add src/hooks/useFormState.js
```

这样Claude能看到完整的代码关系，而不是零散的片段。当你尝试多个假设时，Claude能记住之前的分析。

**技巧2：用指令技巧精准定位问题**

使用结构化的提问方式来精准定位：

```
我的应用在特定条件下出现内存泄漏。

请按以下步骤帮我分析：
1. 首先，列出所有可能导致内存泄漏的原因
2. 然后，根据我的代码特点，排除不可能的原因
3. 最后，给出最可能的3个原因和验证方法

代码片段：[提供代码]
```

这种方式让Claude系统地思考，而不是随意猜测。

**技巧3：用子Agent并行测试多个假设**

当有多个可能的原因时，可以让Claude同时测试多个假设：

```
我的API调用有时成功有时失败。可能的原因有：
1. 网络超时
2. 服务器返回错误
3. 数据格式不匹配

请为每个假设写一个测试用例，我会逐个运行。
```

然后根据测试结果，Claude能快速缩小范围。

---

#### 3. 最佳实践

**如何有效地向Claude描述Bug：**

- 提供最小复现案例（Minimal Reproducible Example）
- 包含完整的错误堆栈跟踪，不要截断
- 说明环境信息（Node版本、浏览器版本等）
- 提供相关的配置文件（package.json、tsconfig.json等）

**提供足够的上下文信息：**

```
不好的描述：
"我的代码不工作"

好的描述：
"在Node 18.0.0 + Express 4.18.0 环境下，
POST /api/users 端点在处理包含特殊字符的用户名时返回500错误。
错误日志显示：SyntaxError: Unexpected token in JSON at position 42"
```

**常见的调试陷阱：**

- 只提供错误信息，不提供代码 → Claude无法定位问题
- 提供过多无关代码 → Claude难以找到关键部分
- 没有说明复现步骤 → Claude无法验证修复
- 隐藏敏感信息导致信息不完整 → 影响诊断准确性

**如何验证修复是否正确：**

修复后，不要只测试一次，要：

1. 测试原始Bug场景是否真的解决了
2. 测试边界情况（空值、极限值、特殊字符等）
3. 测试是否引入了新的Bug（回归测试）
4. 检查性能是否受到影响

```
修复前后对比：
- 原始Bug：用户名包含"@"时崩溃
- 修复后：测试了"@"、"#"、"$"等特殊字符
- 边界测试：空字符串、超长字符串、Unicode字符
- 性能：验证修复没有增加API响应时间
```

### 生成文档

#### 1. 基础解法：让Claude根据代码生成规范文档

最直接的方式是粘贴代码，让Claude生成相应的注释和README。

**提问示例：**

```
请为这个函数生成规范的JSDoc注释：

function calculateDiscount(price, quantity, customerType) {
  if (customerType === 'VIP' && quantity > 10) {
    return price * quantity * 0.85;
  }
  if (quantity > 5) {
    return price * quantity * 0.90;
  }
  return price * quantity;
}

要求：
- 包含参数说明、返回值、使用示例
- 遵循JSDoc格式
```

**生成README示例：**

```
我的项目结构如下：
- /src/api：所有API调用
- /src/components：React组件
- /src/utils：工具函数
- /src/hooks：自定义hooks

能帮我生成一个清晰的README吗？包括项目描述、快速开始、文件结构说明。
```

---

#### 2. 高级技巧

**技巧1：在CLAUDE.md中定义文档规范**

在项目根目录创建或编辑CLAUDE.md，定义你的文档标准：

```
# 项目文档规范

## 代码注释
- 使用JSDoc格式
- 每个公开函数必须有说明
```

### Context管理

#### 1. 什么是Context，为什么重要

**Context**是Claude能记住的对话历史。在Claude Code中，你的每条消息、每个文件引用、每个回答都会被保存在Context中。Context的大小直接影响Claude的回答质量：

- **Context充足**：Claude能看到完整的代码关系、之前的分析、你的偏好设置
- **Context混乱**：无关信息堆积，Claude容易给出不相关的建议
- **Context过期**：旧的信息没有更新，Claude基于过时的假设做决策

---

#### 2. 如何有效地使用Context

**技巧1：用`/clear`清空不需要的历史**

当你切换到完全不同的任务时，清空旧的Context：

```
/clear
```

这样Claude不会被之前的讨论干扰。比如从调试Bug切换到写文档时，清空Context能让Claude更专注。

**技巧2：用`@file`引用关键文件而不是复制粘贴**

不要把整个文件内容复制到消息里，而是用`@file`引用：

```
我想理解@src/utils/validation.js中的逻辑
```

这样Claude能看到文件内容，但不会浪费Context空间。

**技巧3：用CLAUDE.md保存项目级别的信息**

把项目的通用信息写在CLAUDE.md中，而不是每次都重复说：

```
# 项目信息
- 使用TypeScript + React
- 代码风格：Prettier + ESLint
- 测试框架：Jest
```

Claude会自动读取这个文件，你不需要每次都解释。

---

#### 3. Context的常见陷阱

**陷阱1：堆积太多无关信息导致回答变差**

如果你在同一个对话中讨论了10个不同的问题，Claude的回答会变得模糊。解决方案：定期清空Context，为新任务开始新对话。

**陷阱2：忘记更新过时的context**

你说"我的项目用React 16"，但实际已经升级到React 18。Claude会基于过时信息给建议。解决方案：定期检查CLAUDE.md，确保信息是最新的。

**陷阱3：跨项目混淆context**

在项目A中讨论了代码风格，然后切换到项目B。Claude可能会混淆两个项目的约定。解决方案：切换项目时用`/clear`清空Context。

### CLAUDE.md定制

#### 1. CLAUDE.md的作用和语法

**CLAUDE.md**是放在项目根目录的特殊文件，Claude Code会在每次对话开始时自动读取它。它的作用是告诉Claude这个项目的规则和偏好。

基本语法很简单，就是Markdown格式的纯文本：

```
# 项目信息
- 使用TypeScript + React 18
- 代码风格：Prettier + ESLint

## 代码约定
- 函数名用camelCase
- 常量用UPPER_SNAKE_CASE
- 组件用PascalCase

## 测试要求
- 所有公开函数必须有单元测试
- 测试框架：Jest
```

Claude会自动理解这些信息，不需要特殊的语法。

---

#### 2. 常见的定制场景

**场景1：定义代码风格和约定**

```
## 代码风格
- 使用2空格缩进
- 字符串用单引号
- 行长限制80字符
- 禁止使用var，只用const/let
```

这样Claude生成的代码会自动遵循这些规则。

**场景2：指定审查标准**

```
## 代码审查标准
- 必须有错误处理
- 性能关键路径需要注释说明复杂度
- 所有API调用必须有超时设置
```

当你让Claude审查代码时，它会按这些标准检查。

**场景3：设置项目特定的术语和缩写**

```
## 项目术语
- API：后端接口
- UI：用户界面
- DAL：数据访问层
- BFF：后端即服务
```

这样Claude不会误解你的缩写。

---

#### 3. 如何写出有效的CLAUDE.md

**原则1：简洁明确，避免冗长**

不要写成一本书。CLAUDE.md应该是快速参考，而不是详细文档。

```
好的：
- 使用TypeScript
- 测试框架：Jest

不好的：
我们的项目使用TypeScript作为编程语言，因为它提供了类型安全...
```

**原则2：包含具体例子**

抽象的规则Claude可能理解不准确。加上例子会更清楚：

```
## 错误处理
✓ 好的例子：
try {
  const data = await fetchData();
} catch (error) {
  logger.error('Failed to fetch', error);
  throw new CustomError('Data fetch failed');
}

✗ 避免：
try {
  const data = await fetchData();
} catch (e) {
  console.log(e);
}
```

**原则3：定期更新**

当项目规则改变时，更新CLAUDE.md。比如从Jest切换到Vitest，要立即更新。

### Prompt技巧

#### 1. 好Prompt的三个特征

**清晰**：Claude不需要猜测你想要什么。

**具体**：有范围、有标准、有格式要求。

**有上下文**：Claude知道背景，不需要额外解释。

一个对比：

```
模糊的Prompt：
帮我优化这段代码

清晰的Prompt：
帮我优化这段代码的性能。当前的瓶颈是数据库查询太多，
目标是把API响应时间从500ms降到100ms以内。
保持现有的功能不变，只优化性能。
```

---

#### 2. 常见的Prompt模式

**模式1：角色扮演**

让Claude扮演特定角色，能获得更专业的视角：

```
你是一个有10年经验的安全工程师。请审查以下代码，
重点关注SQL注入、XSS攻击等安全漏洞：
[代码]
```

**模式2：对比分析**

让Claude对比两个方案，能获得客观的分析：

```
我有两个实现方案：

方案A：[代码A]
方案B：[代码B]

请从性能、可读性、可维护性三个维度对比这两个方案，
并说明在我们的场景下（高并发、数据量大）推荐哪个。
```

**模式3：分步完成**

把复杂任务分步，让Claude一步一步完成：

```
我需要你分三步完成这个任务：
1. 先分析现有代码的问题
2. 提出重构方案
3. 实现重构后的代码

每步完成后等我确认再进行下一步。
```

---

#### 3. 如何迭代改进Prompt

**步骤1：从Claude的回答判断是否理解了意图**

如果Claude的回答方向不对，说明Prompt有歧义。看Claude的回答，找出它误解的地方。

**步骤2：逐步添加约束条件**

不要一开始就写超长的Prompt。先简单提问，根据回答添加约束：

```
第一次：帮我写一个用户登录函数
第二次：好的，但是需要支持JWT token，不要用session
第三次：还需要加上速率限制，每个IP每分钟最多5次尝试
```

**步骤3：用具体例子替代抽象描述**

```
抽象：生成符合团队风格的注释

具体：生成这种风格的注释：
/**
 * 验证用户登录凭据
 * @param {string} username - 用户名
 * @param {string} password - 明文密码
 * @returns {Promise<User>} 验证成功的用户对象
 * @throws {AuthError} 当凭据无效时抛出
 */
```
