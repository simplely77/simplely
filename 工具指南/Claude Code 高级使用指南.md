## 🔧 日常编码场景

### 理解陌生代码

#### 1. 基础解法

最直接的方法是让 Claude 逐行解释代码。适合场景：接手遗留代码、阅读开源库、理解同事写的复杂逻辑。

**提问示例：**

```
请逐行解释这段代码的作用，重点说明每个函数的输入输出和副作用。

[粘贴代码]
```

```
这里的 reduce 调用是什么意思？为什么初始值是空对象？
```

```
这段代码的核心逻辑是什么？用一句话总结它在做什么。
```

适用场景：代码片段较短（< 200行）、问题明确、不需要理解跨文件依赖。

---

#### 2. 高级技巧

**技巧1：用 @file 引用建立完整上下文**

当陌生代码涉及多个文件时，单独粘贴片段往往无法理解全貌。使用 `@文件名` 引用让 Claude 同时看到相关文件。

```bash
# 引用多个相关文件
@src/auth/middleware.ts @src/types/user.ts
"这个中间件的鉴权逻辑是怎么工作的？token 是从哪里来的？"

# 引用入口文件理解整体结构
@src/index.ts @src/router.ts
"从入口开始，请梳理一个请求的完整处理链路。"
```

**技巧2：用分层提问管理 Context 复杂度**

面对大型陌生代码库时，采用「先宏观后微观」的分层策略，每次对话聚焦一个层次。

```bash
# 第一轮：理解整体架构（保持 context 干净）
"这个项目的目录结构是怎样的？各个模块分别负责什么？"

# 第二轮（新对话）：深入某个模块
@src/payment/ "支付模块的核心流程是什么？从用户点击支付到回调完成，经过哪些关键步骤？"

# 第三轮：聚焦具体问题
@src/payment/processor.ts "processPayment 函数中，为什么要在第47行做两次重试？这个设计意图是什么？"
```

**技巧3：在 CLAUDE.md 中预设代码库背景**

如果你长期需要理解某个代码库，在项目根目录创建 `CLAUDE.md`，让 Claude 每次都带着正确背景知识读代码。

```markdown
## 项目背景
这是一个电商后台服务，基于 NestJS + TypeORM，数据库是 PostgreSQL。

## 核心概念
- Order：订单实体，状态机驱动（pending → paid → shipped → completed）
- SKU：库存单元，与 Product 是多对一关系
- 所有金额字段单位为「分」，避免浮点数精度问题

## 代码约定
- `*Service` 文件包含业务逻辑
- `*Repository` 文件包含数据库操作
- 错误统一用 `AppException` 抛出，不要用原生 Error

## 注意事项
- legacy/ 目录下是旧代码，逻辑可能有问题，不要作为参考
- 凡是有 TODO(tech-debt) 注释的地方都是已知技术债
```

**技巧4：让 Claude 生成代码地图**

对于完全陌生的大型代码库，先让 Claude 生成一份「代码地图」作为后续提问的索引。

```bash
# 生成函数调用关系
@src/ "请列出这个模块中所有公开函数及其调用关系，用树状结构展示。"

# 识别关键路径
"如果我要修改用户注册逻辑，哪些文件会受到影响？请列出需要关注的文件清单。"

# 生成数据流向描述
"请描述数据从 API 入口到数据库写入的完整流向，标出每个环节做了什么转换。"
```

---

#### 3. 最佳实践

- **先问「做什么」，再问「怎么做」**：先理解函数的业务意图，再理解实现细节。直接问实现细节容易迷失在代码噪音中。

- **带着具体问题读代码**：「这个函数为什么要加锁？」比「解释这段代码」能得到更有价值的回答。泛泛的问题得到泛泛的答案。

- **说出你的困惑点**：「这里用了 Proxy，我不太熟悉这个模式，请结合这段代码解释它解决了什么问题」比「解释 Proxy」更有针对性。

- **顺带问「危险区域」**：理解陌生代码时，附带问「这段代码有哪些潜在风险或容易踩坑的地方？」提前了解雷区。

- **用角色扮演获得更深层解释**：「请以这段代码的原作者身份解释，你当时为什么这样设计，有没有考虑过其他方案？」有时能得到更有洞察力的分析。

- **常见陷阱——Context 污染**：在同一对话里理解多段无关代码，会导致 Claude 混淆上下文。理解不同模块时，开启新会话保持 context 干净。

- **常见陷阱——过度信任解释**：Claude 可能对代码意图做出「合理但错误」的推断，特别是当代码本身有 bug 或违反常规模式时。对关键逻辑，用测试或实际运行验证 Claude 的解释。

### 安全重构

#### 1. 基础解法：分析优先，决策权在你

重构前最常见的问题是"一不小心就改坏了"。解决方案很简单：**让Claude先分析，再提建议，你来决定是否执行**。

**提问示例：**

```
我想重构这个函数，但很担心改坏。能帮我分析一下吗？

[粘贴函数代码]

请不要直接改代码，先告诉我：
1. 这个函数的核心逻辑是什么
2. 有哪些潜在的重构方向
3. 每个方向的风险是什么
```

**关键原则：暂时不要动代码**。让Claude充分分析后，你理解了风险和收益，再决定是否让它执行改动。

---

#### 2. 高级技巧

**技巧1：先写测试，再重构（TDD保护网）**

在重构前为现有代码写测试，这样重构后可以验证行为没变。

```
我想重构这个函数。首先，能帮我写一些测试用例吗？

[粘贴函数代码]

请写出能覆盖以下场景的测试：
- 正常输入
- 边界情况
- 异常情况

不要改函数本身，只写测试。
```

测试写好后，运行测试确保通过，然后开始重构。每次改动后都运行测试，确保行为一致。

**技巧2：小步重构，逐步提交**

不要一次性大改，而是分成多个小步骤，每步都是一个独立的改进。

```
我想分步重构这个函数。第一步，能帮我提取这个逻辑块成为一个独立函数吗？

[粘贴代码]

只做这一个改动，保持其他部分不变。
```

每完成一步，就提交一个commit。这样如果某一步出问题，可以轻松回滚。

**技巧3：用子Agent对比方案**

当有多个重构方案时，让Claude分别实现它们，然后对比优缺点。

```
这个函数有两种重构方案：
1. 使用策略模式
2. 使用工厂模式

能分别给我展示这两种方案的实现吗？然后对比它们的优缺点。
```

这样你可以看到具体代码，而不是抽象的描述，做出更明智的选择。

---

#### 3. 最佳实践

- **重构前先有测试**：没有测试的重构是盲目的。即使是简单函数，也值得写几个测试用例。

- **明确约束条件**：告诉Claude你的约束（比如"不能改API"、"必须保持向后兼容"），这样建议会更贴切。

- **小步前进**：每次改动一个小东西，运行测试，提交。不要试图一次性解决所有问题。

- **让Claude解释每个改动**：不要盲目接受建议。要求Claude解释为什么这样改更好，这样你才能学到东西。

**完整工作流示例：**

```
第一步：分析
"这个函数有什么问题吗？"

第二步：写测试
"能帮我写测试吗？"

第三步：小步改进
"第一步，能提取这个逻辑块吗？"

第四步：验证
"测试还通过吗？"

第五步：下一步
"还有其他改进空间吗？"
```

这个流程虽然看起来慢，但实际上是最快的——因为你不会走弯路，也不会改坏代码。

### 代码审查

#### 1. 基础解法

最直接的方式是将代码粘贴给Claude，并明确指出审查的重点：

\`\`\`
我想让你审查这段代码，重点关注：
1. 是否有明显的性能问题
2. 错误处理是否完善
3. 代码可读性

代码如下：
[粘贴代码]
\`\`\`

或者针对特定方面的审查：

\`\`\`
请审查这个函数的安全性，特别是输入验证和边界情况处理。
\`\`\`

**重点关注的方面**：
- **性能**：算法复杂度、不必要的循环、内存泄漏
- **安全**：输入验证、SQL注入、XSS、权限检查
- **可读性**：命名、注释、代码结构
- **可维护性**：重复代码、魔法数字、依赖关系
- **错误处理**：异常捕获、边界情况、日志记录

---

#### 2. 高级技巧

**技巧1：用CLAUDE.md定制审查标准**

在项目根目录创建 \`CLAUDE.md\`，定义团队的代码审查标准：

\`\`\`markdown
# 代码审查标准

## 必检项
- [ ] 所有公开API都有文档注释
- [ ] 错误处理完善（try-catch或Result类型）
- [ ] 没有TODO/FIXME注释
- [ ] 单元测试覆盖率 > 80%

## 性能检查
- [ ] 没有N+1查询问题
- [ ] 没有不必要的深拷贝
- [ ] 循环内没有重复计算

## 安全检查
- [ ] 用户输入都经过验证
- [ ] 敏感信息不在日志中
- [ ] 没有硬编码的密钥
\`\`\`

然后告诉Claude：

\`\`\`
请按照CLAUDE.md中定义的标准审查这段代码。
\`\`\`

**技巧2：用hooks自动化审查**

在 \`.git/hooks/pre-commit\` 中集成Claude审查：

\`\`\`bash
#!/bin/bash
# 获取staged的代码
git diff --cached > staged.diff

# 调用Claude Code进行审查
claude code review staged.diff --check-security --check-performance

# 如果审查失败，阻止提交
if [ $? -ne 0 ]; then
  echo "代码审查未通过，请修复问题后重试"
  exit 1
fi
\`\`\`

**技巧3：针对性审查**

根据不同场景进行专项审查。

**安全审查：**
\`\`\`
请从安全角度审查这段代码，特别检查：
- 输入验证
- 认证/授权
- 数据加密
- SQL注入风险
\`\`\`

**性能审查：**
\`\`\`
请分析这段代码的性能，包括：
- 时间复杂度
- 空间复杂度
- 是否有可优化的地方
- 建议的优化方案
\`\`\`

**可读性审查：**
\`\`\`
请评估代码的可读性和可维护性：
- 变量/函数命名是否清晰
- 是否需要更多注释
- 代码结构是否合理
- 是否有重复代码可以提取
\`\`\`

---

#### 3. 最佳实践

**审查的关键点**：
- 从业务逻辑开始，再看技术细节
- 关注边界情况和异常路径
- 检查是否遵循项目的编码规范
- 验证是否有测试覆盖
- 确认文档和注释的准确性

**常见问题类型**：
- 空指针异常：未检查null/undefined
- 资源泄漏：文件/连接未正确关闭
- 并发问题：竞态条件、死锁
- 性能问题：不必要的循环、重复计算
- 安全漏洞：输入验证不足、权限检查缺失

**如何有效地与Claude协作进行审查**：
- 提供完整的上下文（相关的依赖、配置、测试）
- 明确审查的目标和优先级
- 对Claude的建议进行讨论，而不是盲目接受
- 将审查反馈转化为团队的编码规范
- 定期更新审查标准，基于实际发现的问题

### 调试Bug

#### 1. 基础解法

当遇到Bug时，最有效的方式是向Claude描述现象，让Claude帮助猜测原因。

**如何描述Bug：**

```
我的应用在用户点击"提交"按钮时崩溃了。
错误信息是：TypeError: Cannot read property 'id' of undefined
这发生在处理表单数据的时候。
```

这样的描述包含了三个关键信息：
- 什么时候发生（用户点击提交按钮）
- 错误信息是什么（TypeError）
- 在哪个流程中（表单数据处理）

**提问示例：**

```
我的React组件在初始化时报错：
Error: setState called on unmounted component

代码是这样的：
useEffect(() => {
  fetchData().then(data => {
    setState(data);  // 这里报错
  });
}, []);

这个组件在某些情况下会快速卸载。
```

**有效描述Bug的要点：**
- 提供完整的错误堆栈跟踪
- 说明复现步骤（如何让Bug出现）
- 提供相关的代码片段
- 说明预期行为 vs 实际行为

---

#### 2. 高级技巧

**技巧1：用Context管理保留调试历史**

在调试复杂Bug时，保留完整的对话历史很重要。使用Claude Code的Context功能：

```
/context add src/components/Form.jsx
/context add src/utils/validation.js
/context add src/hooks/useFormState.js
```

这样Claude能看到完整的代码关系，而不是零散的片段。当你尝试多个假设时，Claude能记住之前的分析。

**技巧2：用指令技巧精准定位问题**

使用结构化的提问方式来精准定位：

```
我的应用在特定条件下出现内存泄漏。

请按以下步骤帮我分析：
1. 首先，列出所有可能导致内存泄漏的原因
2. 然后，根据我的代码特点，排除不可能的原因
3. 最后，给出最可能的3个原因和验证方法

代码片段：[提供代码]
```

这种方式让Claude系统地思考，而不是随意猜测。

**技巧3：用子Agent并行测试多个假设**

当有多个可能的原因时，可以让Claude同时测试多个假设：

```
我的API调用有时成功有时失败。可能的原因有：
1. 网络超时
2. 服务器返回错误
3. 数据格式不匹配

请为每个假设写一个测试用例，我会逐个运行。
```

然后根据测试结果，Claude能快速缩小范围。

---

#### 3. 最佳实践

**如何有效地向Claude描述Bug：**

- 提供最小复现案例（Minimal Reproducible Example）
- 包含完整的错误堆栈跟踪，不要截断
- 说明环境信息（Node版本、浏览器版本等）
- 提供相关的配置文件（package.json、tsconfig.json等）

**提供足够的上下文信息：**

```
不好的描述：
"我的代码不工作"

好的描述：
"在Node 18.0.0 + Express 4.18.0 环境下，
POST /api/users 端点在处理包含特殊字符的用户名时返回500错误。
错误日志显示：SyntaxError: Unexpected token in JSON at position 42"
```

**常见的调试陷阱：**

- 只提供错误信息，不提供代码 → Claude无法定位问题
- 提供过多无关代码 → Claude难以找到关键部分
- 没有说明复现步骤 → Claude无法验证修复
- 隐藏敏感信息导致信息不完整 → 影响诊断准确性

**如何验证修复是否正确：**

修复后，不要只测试一次，要：

1. 测试原始Bug场景是否真的解决了
2. 测试边界情况（空值、极限值、特殊字符等）
3. 测试是否引入了新的Bug（回归测试）
4. 检查性能是否受到影响

```
修复前后对比：
- 原始Bug：用户名包含"@"时崩溃
- 修复后：测试了"@"、"#"、"$"等特殊字符
- 边界测试：空字符串、超长字符串、Unicode字符
- 性能：验证修复没有增加API响应时间
```

## 📚 学习新技术场景

### 上手新框架

#### 1. 基础解法：让Claude扮演「框架导师」

学习新框架最高效的方式是让Claude陪你一步步入门，而不是自己啃文档。明确告诉Claude你的背景和目标，让它制定学习路径。

**提问示例：**

```
我是一个有3年Python后端经验的开发者，现在需要学习React。
请扮演我的框架导师，帮我搭建一个Todo List应用。
要求：
1. 每一步都解释「为什么」这样做
2. 对比Python/后端的思维方式说明区别
3. 先带我做最小可运行版本，再逐步扩展
```

---

#### 2. 高级技巧

**技巧1：对比学习法**

利用你已有的知识加速理解。让Claude把新框架的概念对比你熟悉的语言或框架来解释。

```
我熟悉Vue 2，现在学React。能用Vue的概念对比解释以下React概念吗？
- useState vs data()
- useEffect vs mounted/watch
- Context vs Vuex
```

**技巧2：用CLAUDE.md锁定学习约束**

在项目根目录创建 `CLAUDE.md`，声明学习目标和约束，让Claude始终在正确轨道上辅助你。

```markdown
# 学习约束
- 只使用React官方API，不引入第三方库
```

### Context管理

#### 1. 什么是Context，为什么重要

**Context**是Claude能记住的对话历史。在Claude Code中，你的每条消息、每个文件引用、每个回答都会被保存在Context中。Context的大小直接影响Claude的回答质量：

- **Context充足**：Claude能看到完整的代码关系、之前的分析、你的偏好设置
- **Context混乱**：无关信息堆积，Claude容易给出不相关的建议
- **Context过期**：旧的信息没有更新，Claude基于过时的假设做决策

---

#### 2. 如何有效地使用Context

**技巧1：用`/clear`清空不需要的历史**

当你切换到完全不同的任务时，清空旧的Context：

```
/clear
```

这样Claude不会被之前的讨论干扰。比如从调试Bug切换到写文档时，清空Context能让Claude更专注。

**技巧2：用`@file`引用关键文件而不是复制粘贴**

不要把整个文件内容复制到消息里，而是用`@file`引用：

```
我想理解@src/utils/validation.js中的逻辑
```

这样Claude能看到文件内容，但不会浪费Context空间。

**技巧3：用CLAUDE.md保存项目级别的信息**

把项目的通用信息写在CLAUDE.md中，而不是每次都重复说：

```
# 项目信息
- 使用TypeScript + React
- 代码风格：Prettier + ESLint
- 测试框架：Jest
```

Claude会自动读取这个文件，你不需要每次都解释。

---

#### 3. Context的常见陷阱

**陷阱1：堆积太多无关信息导致回答变差**

如果你在同一个对话中讨论了10个不同的问题，Claude的回答会变得模糊。解决方案：定期清空Context，为新任务开始新对话。

**陷阱2：忘记更新过时的context**

你说"我的项目用React 16"，但实际已经升级到React 18。Claude会基于过时信息给建议。解决方案：定期检查CLAUDE.md，确保信息是最新的。

**陷阱3：跨项目混淆context**

在项目A中讨论了代码风格，然后切换到项目B。Claude可能会混淆两个项目的约定。解决方案：切换项目时用`/clear`清空Context。

### CLAUDE.md定制

#### 1. CLAUDE.md的作用和语法

**CLAUDE.md**是放在项目根目录的特殊文件，Claude Code会在每次对话开始时自动读取它。它的作用是告诉Claude这个项目的规则和偏好。

基本语法很简单，就是Markdown格式的纯文本：

```
# 项目信息
- 使用TypeScript + React 18
- 代码风格：Prettier + ESLint

## 代码约定
- 函数名用camelCase
- 常量用UPPER_SNAKE_CASE
- 组件用PascalCase

## 测试要求
- 所有公开函数必须有单元测试
- 测试框架：Jest
```

Claude会自动理解这些信息，不需要特殊的语法。

---

#### 2. 常见的定制场景

**场景1：定义代码风格和约定**

```
## 代码风格
- 使用2空格缩进
- 字符串用单引号
- 行长限制80字符
- 禁止使用var，只用const/let
```

这样Claude生成的代码会自动遵循这些规则。

**场景2：指定审查标准**

```
## 代码审查标准
- 必须有错误处理
- 性能关键路径需要注释说明复杂度
- 所有API调用必须有超时设置
```

当你让Claude审查代码时，它会按这些标准检查。

**场景3：设置项目特定的术语和缩写**

```
## 项目术语
- API：后端接口
- UI：用户界面
- DAL：数据访问层
- BFF：后端即服务
```

这样Claude不会误解你的缩写。

---

#### 3. 如何写出有效的CLAUDE.md

**原则1：简洁明确，避免冗长**

不要写成一本书。CLAUDE.md应该是快速参考，而不是详细文档。

```
好的：
- 使用TypeScript
- 测试框架：Jest

不好的：
我们的项目使用TypeScript作为编程语言，因为它提供了类型安全...
```

**原则2：包含具体例子**

抽象的规则Claude可能理解不准确。加上例子会更清楚：

```
## 错误处理
✓ 好的例子：
try {
  const data = await fetchData();
} catch (error) {
  logger.error('Failed to fetch', error);
  throw new CustomError('Data fetch failed');
}

✗ 避免：
try {
  const data = await fetchData();
} catch (e) {
  console.log(e);
}
```

**原则3：定期更新**

当项目规则改变时，更新CLAUDE.md。比如从Jest切换到Vitest，要立即更新。

### Prompt技巧

#### 1. 好Prompt的三个特征

**清晰**：Claude不需要猜测你想要什么。

**具体**：有范围、有标准、有格式要求。

**有上下文**：Claude知道背景，不需要额外解释。

一个对比：

```
模糊的Prompt：
帮我优化这段代码

清晰的Prompt：
帮我优化这段代码的性能。当前的瓶颈是数据库查询太多，
目标是把API响应时间从500ms降到100ms以内。
保持现有的功能不变，只优化性能。
```

---

#### 2. 常见的Prompt模式

**模式1：角色扮演**

让Claude扮演特定角色，能获得更专业的视角：

```
你是一个有10年经验的安全工程师。请审查以下代码，
重点关注SQL注入、XSS攻击等安全漏洞：
[代码]
```

**模式2：对比分析**

让Claude对比两个方案，能获得客观的分析：

```
我有两个实现方案：

方案A：[代码A]
方案B：[代码B]

请从性能、可读性、可维护性三个维度对比这两个方案，
并说明在我们的场景下（高并发、数据量大）推荐哪个。
```

**模式3：分步完成**

把复杂任务分步，让Claude一步一步完成：

```
我需要你分三步完成这个任务：
1. 先分析现有代码的问题
2. 提出重构方案
3. 实现重构后的代码

每步完成后等我确认再进行下一步。
```

---

#### 3. 如何迭代改进Prompt

**步骤1：从Claude的回答判断是否理解了意图**

如果Claude的回答方向不对，说明Prompt有歧义。看Claude的回答，找出它误解的地方。

**步骤2：逐步添加约束条件**

不要一开始就写超长的Prompt。先简单提问，根据回答添加约束：

```
第一次：帮我写一个用户登录函数
第二次：好的，但是需要支持JWT token，不要用session
第三次：还需要加上速率限制，每个IP每分钟最多5次尝试
```

**步骤3：用具体例子替代抽象描述**

```
抽象：生成符合团队风格的注释

具体：生成这种风格的注释：
/**
 * 验证用户登录凭据
 * @param {string} username - 用户名
 * @param {string} password - 明文密码
 * @returns {Promise<User>} 验证成功的用户对象
 * @throws {AuthError} 当凭据无效时抛出
 */
```

### 多文件操作

#### 1. 风险识别

大范围修改（比如一次改20个文件）容易出错的原因：

- **没看到全局影响**：改了A文件，可能破坏了与B、C文件的交互
- **难以逐一验证**：修改太多，测试不周全
- **无法快速回滚**：出问题了，难以找出是哪个文件改坏了

---

#### 2. 安全的操作流程

**流程1：先在小范围测试**

不要一次性修改10个文件。改1-2个文件试试：

```
我想统一重命名项目中所有的错误类。
能先在src/errors/下的3个文件中试验一下吗？
看成功后再批量应用。
```

**流程2：用git分支隔离**

创建独立的分支做大范围修改，这样随时可以回到原始状态：

```bash
git checkout -b refactor/rename-errors
```

修改过程中，如果发现问题，可以快速回滚：

```bash
git reset --hard HEAD
```

**流程3：频繁提交，便于回滚**

不要改完所有文件再提交一次。每改完一批文件就提交一次，粒度越小越安全：

```bash
git add src/errors/AppError.js
git commit -m 'Rename AppError to CustomError'
```

---

#### 3. 如何验证修改的正确性

**验证1：让Claude生成测试用例**

```
我改动了数据库查询层的所有函数名。
能帮我写一些测试用例来验证功能没有破坏吗？
关键的测试场景有：正常查询、错误处理、并发查询。
```

**验证2：运行现有测试**

```bash
npm test
```

确保原有的测试都通过。如果有失败，说明修改有问题。

**验证3：手动spot check关键文件**

```
能帮我检查这两个文件吗？确保修改都正确了：
@src/api/userApi.js
@src/api/productApi.js
```


### 子Agent

#### 1. 什么时候用子Agent

**子Agent**是指让Claude同时处理多个独立的任务，而不是一个接一个地处理。什么时候值得用子Agent？

**场景1：任务可以独立完成**

如果任务A和任务B没有依赖关系，可以并行处理：

```
我需要：
1. 为这个模块写单元测试
2. 为那个模块写集成测试

这两个任务独立，能同时做吗？
```

**场景2：需要对比多个方案**

```
我有三种实现方案，能分别给我展示代码吗？
然后对比它们的优缺点。
```

**场景3：多个文件的独立修改**

```
我需要在这5个文件中都添加错误处理。
能分别为每个文件生成修改方案吗？
```

---

#### 2. 如何有效地分解任务

**原则1：每个子任务有明确的输入和输出**

```
不清楚的分解：
- 优化代码
- 改进性能

清楚的分解：
- 输入：src/api/user.js
  输出：性能优化后的代码，响应时间从500ms降到100ms
- 输入：src/utils/validation.js
  输出：添加类型检查和错误处理的代码
```

**原则2：子任务之间尽量独立**

如果子任务A的输出是子任务B的输入，就不能并行。要重新组织任务。

**原则3：提供充分的上下文**

每个子任务都要有足够的信息，不要让Claude去猜测：

```
好的：
任务1：为UserService.js中的login函数写单元测试。
      测试框架是Jest，mock库是sinon。
      需要覆盖：正常登录、密码错误、用户不存在。

不好的：
任务1：写测试
```

---

#### 3. 常见的并行模式

**模式1：多个文件的独立修改**

```
我需要在这3个文件中都添加日志记录：
- src/api/userApi.js
- src/api/productApi.js
- src/api/orderApi.js

能分别为每个文件生成修改方案吗？
```

**模式2：多个测试用例的并行验证**

```
我的API有这些场景需要测试：
1. 正常请求
2. 超时请求
3. 错误响应

能分别为每个场景写一个测试用例吗？
```

**模式3：多个方案的对比实现**

```
这个功能有两种实现方式：
1. 使用递归
2. 使用迭代

能分别实现这两种方式，然后对比性能吗？
```

## 📎 附录：常用命令速查

### Claude Code 快捷键和命令

| 快捷键/命令 | 功能 | 说明 |
|-----------|------|------|
| `@file` | 快速引用文件 | 在提问中直接引用文件路径，无需复制粘贴 |
| `@folder/` | 引用整个文件夹 | 让Claude阅读文件夹中的所有文件 |
| `/clear` | 清空context | 清除对话历史，开始新的对话 |
| `Shift+Enter` | 换行 | 在输入框中换行，不发送消息 |
| `Ctrl/Cmd+K` | 新对话 | 开始一个新的对话 |
| `Ctrl/Cmd+/` | 快捷键帮助 | 查看所有快捷键 |

### 常见的Prompt模式

**1. 角色扮演**
```
你是一个[角色]，请帮我[任务]
```

**2. 对比分析**
```
对比以下两个方案的优缺点：
方案A：[描述]
方案B：[描述]
```

**3. 分步骤执行**
```
请分3个步骤完成[任务]：
1. [第一步]
2. [第二步]
3. [第三步]
```

**4. 代码审查**
```
@file 请对这个文件做代码审查，重点关注：
- 性能问题
- 安全隐患
- 可读性
```

### 最佳实践速查

- **提供背景信息**：告诉Claude你的项目是什么、用什么技术栈
- **明确约束条件**：「不要改接口」「保持向后兼容」
- **要求具体输出**：「生成单元测试」「用Mermaid图表示」
- **验证理解**：让Claude用你的语言重述理解
- **小步前进**：大任务分解成小任务，逐步完成
- **保存有用的context**：在CLAUDE.md中记录项目级别的信息

## 高级 CLI 使用技巧

### 为什么要用 CLI

Claude Code 的 CLI 模式特别适合：
- **快速一次性任务**：不需要打开 GUI，直接在终端完成
- **管道操作**：将命令输出直接传给 Claude 处理
- **脚本自动化**：集成到 shell 脚本、Git hooks、CI/CD 中
- **Headless 模式**：在服务器、容器、远程环境中使用

### 基础 CLI 用法

**直接提问（最简单）**

```bash
# 快速提问
cc "解释这段代码在做什么" < script.js

# 带文件引用
cc "重构这个函数，提取重复逻辑" @src/utils/helper.js

# 多文件分析
cc "这两个文件的依赖关系是什么" @src/api.js @src/db.js
```

**管道输入（最强大）**

```bash
# 将命令输出传给 Claude
git diff | cc "总结这次改动，生成 commit message"

# 分析错误日志
npm test 2>&1 | cc "分析测试失败的原因，给出修复建议"

# 处理复杂输出
docker ps -a | cc "找出占用内存最多的3个容器，解释为什么"

# 代码审查
git diff main...HEAD | cc "按照 CLAUDE.md 的标准审查这些改动"
```

### 高级 CLI 技巧

**技巧1：Headless 模式（无交互）**

适合脚本和自动化场景，Claude 直接返回结果，不需要人工确认：

```bash
# 启用 headless 模式
cc --headless "分析这个函数的时间复杂度" < algorithm.js

# 在 CI/CD 中使用
cc --headless --output-format json "检查代码是否符合规范" @src/**/*.js

# 批量处理
for file in src/*.js; do
  cc --headless "为这个文件生成单元测试" < "$file" > "tests/${file%.js}.test.js"
done
```

**技巧2：输出格式控制**

```bash
# JSON 格式输出（便于脚本解析）
cc --output-format json "列出这个项目的所有 API 端点" @src/routes/

# Markdown 格式（便于文档）
cc --output-format markdown "生成这个模块的 API 文档" @src/api/ > docs/api.md

# 纯文本（默认）
cc --output-format text "解释这段代码"
```

**技巧3：与 Git 工作流集成**

```bash
# Pre-commit hook：自动审查代码
# .git/hooks/pre-commit
#!/bin/bash
git diff --cached | cc --headless "快速审查：是否有明显的 bug、安全问题或性能问题？如果有，列出来；如果没有，只回复 OK"
if [ $? -ne 0 ]; then
  echo "代码审查发现问题，请修复后再提交"
  exit 1
fi

# 生成 commit message
git diff --cached | cc "生成简洁的 commit message（conventional commits 格式）" --print

# 生成 PR 描述
git diff main...HEAD | cc "生成 PR 描述，包括 What/Why/How" > pr-description.md
```

**技巧4：快速代码生成**

```bash
# 生成测试文件
cc "为这个函数生成 Jest 测试用例" < src/utils.js > tests/utils.test.js

# 生成类型定义
cc "为这个 JS 文件生成 TypeScript 类型定义" < api.js > api.d.ts

# 生成文档
cc "生成 JSDoc 注释" < src/module.js --print | pbcopy  # macOS
cc "生成 JSDoc 注释" < src/module.js --print | xclip    # Linux
```

**技巧5：日志分析和调试**

```bash
# 分析应用日志
tail -n 100 app.log | cc "找出错误模式，给出可能的原因"

# 分析性能瓶颈
npm run profile 2>&1 | cc "识别性能瓶颈，给出优化建议"

# 解释复杂错误
cargo build 2>&1 | cc "用简单的语言解释这些编译错误"
```

**技巧6：批量文件处理**

```bash
# 批量重命名（先预览）
ls src/*.js | cc "建议更符合语义的文件名（输出 JSON 格式：{old: new}）"

# 批量添加注释
find src -name "*.js" -exec sh -c '
  cc "为这个文件的导出函数添加 JSDoc 注释" < "$1" > "$1.tmp" && mv "$1.tmp" "$1"
' _ {} \;

# 批量代码审查
find src -name "*.js" | while read file; do
  echo "=== $file ===" >> review.md
  cc "快速审查这个文件" < "$file" >> review.md
done
```

**技巧7：与其他工具链组合**

```bash
# 与 jq 组合处理 JSON
curl https://api.example.com/data | jq '.' | cc "分析这个 API 响应的结构，生成 TypeScript 接口"

# 与 ripgrep 组合
rg "TODO" -A 3 | cc "整理所有 TODO 项，按优先级排序"

# 与 fd 组合
fd -e js -e ts | cc "分析项目结构，生成模块依赖图（Mermaid 格式）"

# 与 gh 组合
gh pr view 123 --json body | cc "总结这个 PR 的主要改动"
```

### CLI 实战场景

**场景1：快速代码审查**

```bash
# 审查当前改动
git diff | cc "代码审查：检查性能、安全、可读性问题" --headless

# 审查特定提交
git show abc123 | cc "这个提交有什么潜在问题？"

# 审查整个 PR
gh pr diff 456 | cc "全面审查这个 PR，给出改进建议"
```

**场景2：自动化文档生成**

```bash
# 生成 README
cc "根据项目结构生成 README" @package.json @src/ > README.md

# 生成 CHANGELOG
git log --oneline main..HEAD | cc "生成 CHANGELOG（Keep a Changelog 格式）" > CHANGELOG.md

# 生成 API 文档
cc "生成完整的 API 文档" @src/api/ --output-format markdown > docs/api.md
```

**场景3：智能搜索和重构**

```bash
# 找出所有使用某个 API 的地方
rg "oldFunction" -l | xargs cc "列出所有调用 oldFunction 的位置，建议如何重构"

# 批量重构
cc "生成重构脚本：将所有 var 改为 const/let" @src/ > refactor.sh

# 依赖分析
cc "分析哪些模块依赖了这个文件" @src/utils/deprecated.js
```

### CLI 最佳实践

**性能优化**

```bash
# 使用 --print 避免交互确认（适合简单查询）
cc --print "这个函数的时间复杂度是多少" < algo.js

# 限制输入大小（避免超过 token 限制）
head -n 100 large-file.log | cc "分析前100行日志"

# 使用缓存（对于重复查询）
cc --cache "解释这个配置文件" < config.yaml
```

**安全注意事项**

```bash
# 不要将敏感信息传给 Claude
git diff | grep -v "password\|secret\|token" | cc "审查代码"

# 使用 --dry-run 预览操作（如果支持）
cc --dry-run "重构这个文件" < src/api.js

# 审查 Claude 生成的脚本后再执行
cc "生成清理脚本" > clean.sh
# 先检查 clean.sh 的内容
cat clean.sh
# 确认无误后再执行
bash clean.sh
```

**调试 CLI 问题**

```bash
# 启用详细日志
cc --verbose "分析代码" < file.js

# 查看 API 调用详情
```
