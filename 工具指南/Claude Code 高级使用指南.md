### 安全重构

#### 1. 基础解法：分析优先，决策权在你

重构前最常见的问题是"一不小心就改坏了"。解决方案很简单：**让Claude先分析，再提建议，你来决定是否执行**。

**提问示例：**

```
我想重构这个函数，但很担心改坏。能帮我分析一下吗？

[粘贴函数代码]

请不要直接改代码，先告诉我：
1. 这个函数的核心逻辑是什么
2. 有哪些潜在的重构方向
3. 每个方向的风险是什么
```

**关键原则：暂时不要动代码**。让Claude充分分析后，你理解了风险和收益，再决定是否让它执行改动。

---

#### 2. 高级技巧

**技巧1：先写测试，再重构（TDD保护网）**

在重构前为现有代码写测试，这样重构后可以验证行为没变。

```
我想重构这个函数。首先，能帮我写一些测试用例吗？

[粘贴函数代码]

请写出能覆盖以下场景的测试：
- 正常输入
- 边界情况
- 异常情况

不要改函数本身，只写测试。
```

测试写好后，运行测试确保通过，然后开始重构。每次改动后都运行测试，确保行为一致。

**技巧2：小步重构，逐步提交**

不要一次性大改，而是分成多个小步骤，每步都是一个独立的改进。

```
我想分步重构这个函数。第一步，能帮我提取这个逻辑块成为一个独立函数吗？

[粘贴代码]

只做这一个改动，保持其他部分不变。
```

每完成一步，就提交一个commit。这样如果某一步出问题，可以轻松回滚。

**技巧3：用子Agent对比方案**

当有多个重构方案时，让Claude分别实现它们，然后对比优缺点。

```
这个函数有两种重构方案：
1. 使用策略模式
2. 使用工厂模式

能分别给我展示这两种方案的实现吗？然后对比它们的优缺点。
```

这样你可以看到具体代码，而不是抽象的描述，做出更明智的选择。

---

#### 3. 最佳实践

- **重构前先有测试**：没有测试的重构是盲目的。即使是简单函数，也值得写几个测试用例。

- **明确约束条件**：告诉Claude你的约束（比如"不能改API"、"必须保持向后兼容"），这样建议会更贴切。

- **小步前进**：每次改动一个小东西，运行测试，提交。不要试图一次性解决所有问题。

- **让Claude解释每个改动**：不要盲目接受建议。要求Claude解释为什么这样改更好，这样你才能学到东西。

**完整工作流示例：**

```
第一步：分析
"这个函数有什么问题吗？"

第二步：写测试
"能帮我写测试吗？"

第三步：小步改进
"第一步，能提取这个逻辑块吗？"

第四步：验证
"测试还通过吗？"

第五步：下一步
"还有其他改进空间吗？"
```

这个流程虽然看起来慢，但实际上是最快的——因为你不会走弯路，也不会改坏代码。

### 调试Bug

#### 1. 基础解法

当遇到Bug时，最有效的方式是向Claude描述现象，让Claude帮助猜测原因。

**如何描述Bug：**

```
我的应用在用户点击"提交"按钮时崩溃了。
错误信息是：TypeError: Cannot read property 'id' of undefined
这发生在处理表单数据的时候。
```

这样的描述包含了三个关键信息：
- 什么时候发生（用户点击提交按钮）
- 错误信息是什么（TypeError）
- 在哪个流程中（表单数据处理）

**提问示例：**

```
我的React组件在初始化时报错：
Error: setState called on unmounted component

代码是这样的：
useEffect(() => {
  fetchData().then(data => {
    setState(data);  // 这里报错
  });
}, []);

这个组件在某些情况下会快速卸载。
```

**有效描述Bug的要点：**
- 提供完整的错误堆栈跟踪
- 说明复现步骤（如何让Bug出现）
- 提供相关的代码片段
- 说明预期行为 vs 实际行为

---

#### 2. 高级技巧

**技巧1：用Context管理保留调试历史**

在调试复杂Bug时，保留完整的对话历史很重要。使用Claude Code的Context功能：

```
/context add src/components/Form.jsx
/context add src/utils/validation.js
/context add src/hooks/useFormState.js
```

这样Claude能看到完整的代码关系，而不是零散的片段。当你尝试多个假设时，Claude能记住之前的分析。

**技巧2：用指令技巧精准定位问题**

使用结构化的提问方式来精准定位：

```
我的应用在特定条件下出现内存泄漏。

请按以下步骤帮我分析：
1. 首先，列出所有可能导致内存泄漏的原因
2. 然后，根据我的代码特点，排除不可能的原因
3. 最后，给出最可能的3个原因和验证方法

代码片段：[提供代码]
```

这种方式让Claude系统地思考，而不是随意猜测。

**技巧3：用子Agent并行测试多个假设**

当有多个可能的原因时，可以让Claude同时测试多个假设：

```
我的API调用有时成功有时失败。可能的原因有：
1. 网络超时
2. 服务器返回错误
3. 数据格式不匹配

请为每个假设写一个测试用例，我会逐个运行。
```

然后根据测试结果，Claude能快速缩小范围。

---

#### 3. 最佳实践

**如何有效地向Claude描述Bug：**

- 提供最小复现案例（Minimal Reproducible Example）
- 包含完整的错误堆栈跟踪，不要截断
- 说明环境信息（Node版本、浏览器版本等）
- 提供相关的配置文件（package.json、tsconfig.json等）

**提供足够的上下文信息：**

```
不好的描述：
"我的代码不工作"

好的描述：
"在Node 18.0.0 + Express 4.18.0 环境下，
POST /api/users 端点在处理包含特殊字符的用户名时返回500错误。
错误日志显示：SyntaxError: Unexpected token in JSON at position 42"
```

**常见的调试陷阱：**

- 只提供错误信息，不提供代码 → Claude无法定位问题
- 提供过多无关代码 → Claude难以找到关键部分
- 没有说明复现步骤 → Claude无法验证修复
- 隐藏敏感信息导致信息不完整 → 影响诊断准确性

**如何验证修复是否正确：**

修复后，不要只测试一次，要：

1. 测试原始Bug场景是否真的解决了
2. 测试边界情况（空值、极限值、特殊字符等）
3. 测试是否引入了新的Bug（回归测试）
4. 检查性能是否受到影响

```
修复前后对比：
- 原始Bug：用户名包含"@"时崩溃
- 修复后：测试了"@"、"#"、"$"等特殊字符
- 边界测试：空字符串、超长字符串、Unicode字符
- 性能：验证修复没有增加API响应时间
```
